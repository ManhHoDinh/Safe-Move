## SLIDE 16: DATA PROCESSING
**Layout:** Horizontal pipeline

**Headline:**
Raw Signals → Clean Events

**Visual:** The data processing pipeline showing REAL system design decisions:

```
                    DATA PROCESSING PIPELINE

  ┌──────────────────────────────────────────────────────────┐
  │  RAW INPUT                                               │
  │  • Camera frames (1K fps, RTSP)                          │
  │  • Weather JSON (15-min batches)                         │
  │  • News text (unstructured, multilingual)                │
  │  • User reports (free-text + photos + GPS)               │
  └──────────────┬───────────────────────────────────────────┘
                 ▼
  ┌─────┐  ┌──────────┐  ┌─────────┐  ┌──────────┐  ┌──────┐
  │DEDUP│→│NORMALIZE │→│ GEO-TAG │→│TIME-ALIGN│→│ FUSE │
  │     │  │          │  │         │  │          │  │      │
  │Drop │  │cm/WGS84/ │  │PostGIS  │  │30-second │  │Multi-│
  │dupl-│  │UTC/JSON  │  │spatial  │  │windows   │  │source│
  │icates│  │schema    │  │join to  │  │align     │  │merge │
  │     │  │          │  │road     │  │events    │  │+score│
  └─────┘  └──────────┘  │segments │  └──────────┘  └──────┘
                         └─────────┘
                 ▼
  ┌──────────────────────────────────────────────────────────┐
  │  CLEAN EVENT OUTPUT                                      │
  │  {                                                       │
  │    location: { lat, lng, road_segment_id },              │
  │    sources: ["camera_047", "weather_api", "user_report"],│
  │    severity_estimate: 3,                                 │
  │    confidence: 0.87,                                     │
  │    timestamp: "2026-03-21T14:23:00Z"                     │
  │  }                                                       │
  └──────────────────────────────────────────────────────────┘
```

**Key design decisions shown on slide:**

**Trade-off 1: Stream vs Batch**
- Stream (Flink): Camera, user reports → need < 5s latency
- Batch (Spark): Historical analysis, model retraining → hourly/daily
- "We don't stream everything. We stream what needs to be fast."

**Trade-off 2: Geo-tagging precision**
- PostGIS spatial join maps every event to road segments (not just lat/lng)
- Why: routing penalties must apply to specific road segments, not circular areas
- Cost: 50ms per event for spatial query. Worth it for routing accuracy.

**Trade-off 3: 30-second fusion windows**
- Why 30s? Camera and weather might detect same flood 10-20s apart
- Too short (5s): miss correlated events → duplicate flood points
- Too long (5min): delay in detection → drivers already in flood
- 30s balances dedup vs speed

## SLIDE 17: FLOOD DETECTION + LLM
**Layout:** Split — detection left, LLM right

**Headline:**
Detect with CV. Assess with LLM.

**Visual:**

LEFT: Detection convergence diagram
```
[Camera] ──┐
[Weather] ─┤→ [DETECTION] → Flood Point
[News] ────┤     ENGINE      (lat/lng + confidence)
[Users] ───┘
```

RIGHT: LLM assessment flow
```
INPUT:                    OUTPUT:
"Camera: water visible"   Level: 3/5
"Weather: 85mm/2hr"  →   Depth: ~35cm
"News: District 7..."  → Passable: No (sedans)
"User: knee-deep"        Confidence: 91%
```

**Key insight shown:** "CV detects PRESENCE. LLM assesses SEVERITY. Why? No labeled severity dataset exists. LLM reasons from context — no training data needed."
